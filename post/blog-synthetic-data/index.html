Synthetic Data Generation — Statistically Grounded, Business-Aware

April 5, 2026

Generating synthetic data is easy if you don't care whether it's useful. Sample random numbers within a range, give the columns plausible names, and you have something that looks like a dataset. The problem is that a model trained on that data will learn nothing meaningful, and a data quality test run against it will tell you nothing true.

I built a synthetic data generation system designed around the opposite philosophy: every generated record should be statistically faithful to the real data it represents, every sensitive column should be replaced with privacy-safe equivalents, and the structure of real business segments should be preserved — not averaged out.

Where It Starts: Context, Not Just Schema

Most synthetic data tools take a schema as input: column names and types. This system took more than that. It started with the full context of the business problem — the objective, the domain, the data products the schema represented, the segment rules extracted from the real data by the automated cataloging pipeline, and the PII flags already identified per column.

That richer input is what made the output more useful. A column named "revenue" means something different in a pharmaceutical supply chain dataset than in a consumer electronics dataset. With domain context, the generation could be tuned to produce values in the range and distribution that actually made sense for that business, not just whatever the historical data happened to look like.

How Values Were Generated

Numerical columns were generated by first determining the best-fitting statistical distribution for each column and then using the business context to calibrate the parameters. Rather than blindly copying the historical distribution (which might reflect a biased or unrepresentative sample), this approach used domain knowledge to ask: what should the distribution of this variable look like for this business? The final values were then drawn from that calibrated parametric distribution.

Categorical columns were generated based on reasoned value sets and frequencies: what are the plausible categories for this variable in this domain, and what should their relative proportions be? This avoided the trap of over-representing whatever happened to dominate the training data.

Columns flagged as PII — names, email addresses, phone numbers, national IDs — were replaced entirely with structurally valid but entirely fictional identifiers, locale and format calibrated to match the data domain: a European pharma dataset got European-format names and addresses, not American defaults.

Segment-Aware Generation

This is the feature that most synthetic data generators skip, and it's the most important one for producing data that's actually useful for ML.

Real datasets are not drawn uniformly from a single population. They contain segments — subgroups with different behaviors and different distributions. A customer base has high-value and low-value segments. A loan portfolio has low-risk and high-risk borrower profiles. A product catalog has fast-moving and slow-moving SKUs. If you generate synthetic data from the global average distribution, you erase those segments. A model trained on the result won't learn the conditional patterns that make the real data informative.

The cataloging pipeline I built extracted these segment structures automatically using decision tree models — producing human-readable rules that defined each segment and its statistics. The synthetic data generator used those rules directly: it assigned each synthetic record to a segment first, then drew that record's values from the conditional distributions for that segment. The result was synthetic data that preserved the structure of the real data, not just its marginal statistics.

Evaluating the Output: A Three-Dimensional Quality Score

Every generation run produced a quality score across three dimensions:

Fidelity asked: how statistically similar is the synthetic data to the real data? This covered column-level distribution similarity and whether the correlation structure between columns was preserved.

Utility asked: would a model trained on this synthetic data perform similarly to one trained on real data? Evaluated by training on synthetic and testing on real, then comparing against the reverse — the gap is the utility loss.

Privacy asked: could someone use the synthetic data to infer real values or re-identify real individuals? This checked for cases where synthetic records were too statistically close to specific real records — a sign that the generation had memorized rather than generalized.

Together, these three scores gave the user a clear picture of what they were getting — and what trade-offs existed between fidelity and privacy for their specific dataset.

Why This Matters

Synthetic data is increasingly the answer to two problems that aren't going away: data governance (sharing real customer data with external models or teams is often legally restricted) and training data scarcity (real datasets are frequently too small, too skewed, or too expensive to collect at the required scale). A generator that is distribution-aware, segment-preserving, and privacy-evaluated is production infrastructure — not a demo.

Building it required combining statistical knowledge (distribution families, correlation preservation), LLM-based reasoning (structured parameter generation from domain context), and software engineering (multi-table relationships, async execution, cloud delivery, quality evaluation integration). That intersection — not any single skill in isolation — was what made the feature work.
