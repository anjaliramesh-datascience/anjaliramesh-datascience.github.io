Building an Automated Data Catalog — End to End

March 1, 2024

Most data catalogs are search interfaces maintained by people. Someone uploads a dataset, someone else manually tags it, writes a description, flags which columns are sensitive, and marks what it might be used for. It's slow, it doesn't scale, and it's only as good as the person doing the tagging.

I built something different: a fully automated catalog pipeline that takes raw tabular data — from any source — and produces a rich, structured, ML-ready catalog within minutes. No manual tagging. No human curation after the upload. I owned this feature end to end, from architecture to deployment, and it ran in production at scale serving enterprise clients.

The Pipeline

The moment a dataset arrives, a sequence of automated steps runs in order.

First: figure out what the data actually is. Custom type inference logic assigned each column a semantic type — not just "integer" or "string," but whether a numeric column represented a date stored as a year, a transaction amount, or a count. Getting this right mattered because every downstream step — cleaning, correlation analysis, ML modeling — needed to treat different column types differently.

Second: clean it. Columns that carried no signal — fully empty, zero-variance, ID columns where every row was unique, or categorical columns with too many distinct values to be useful — were dropped automatically. The ML pipeline downstream only saw data that could actually inform a model.

Third: profile it statistically. For every numeric column, I fit multiple statistical distributions to find the best-fitting shape. This dual-purposed: it described the column's behavior for exploration, and it stored the parameters needed to generate realistic synthetic data later.

Fourth: analyze relationships between columns using the statistically appropriate method for each pair — which isn't the same method for every combination. A continuous variable paired with another continuous variable calls for a different test than a categorical variable paired with a continuous one. I implemented type-aware correlation selection so the analysis was actually valid, not just numerically convenient.

Fifth: extract segmentation rules automatically using decision tree models trained per target variable. The output wasn't a model — it was a set of human-readable conditions: the feature combinations that best predicted each outcome in the dataset. Business teams could read these directly without knowing what a decision tree was.

Sixth: run LLM-powered metadata generation across several distinct tasks — writing column descriptions, flagging PII and sensitive data (two separate prompts, deliberately, because combining them hurt precision), identifying the ML problem type, and calling out which columns were likely targets or segmentation variables for downstream modeling.

Seventh: classify every column into a four-level enterprise taxonomy — domain, subdomain, data product, and logical data element — using an AI assistant with access to the full organizational data model. New mappings that didn't exist in the taxonomy were created on the fly and added back, so the system learned with each new dataset.

At scale, this processed thousands of enterprise datasets in production. Each dataset went from raw upload to fully cataloged in approximately five to seven minutes.

What Made It Hard

The honest answer is consistency. Every step had to produce correct output for every combination of column types that could appear in a real enterprise dataset — and enterprise datasets are messy. The type inference had to handle years-as-integers, booleans stored as "Y"/"N" strings, and timestamps in a dozen different formats. The correlation logic had to select the right test at runtime. The segmentation rule extraction had to reverse all the preprocessing transformations to produce readable output instead of encoded feature names and float thresholds.

Each individual step was manageable. Making all of them work correctly together, on data you've never seen before, from clients whose schemas you didn't design — that was the engineering challenge. And it ran in production, reliably, for thousands of datasets.

The LLM components required prompt engineering discipline as well. Running PII detection and sensitive data detection as separate, purpose-built prompts — rather than one combined call — was a deliberate decision made after observing that combined prompts degraded precision. Separate prompts with explicit scope definitions produced more reliable, auditable outputs. That kind of empirical tuning is unglamorous but necessary for production-grade AI components.

What This Enabled Downstream

A fully structured catalog is not just a search index. It's the foundation for everything else: synthetic data generation needs the statistical profiles, auto-ML needs the column type assignments and ML problem classification, use case development needs the domain tagging. The catalog pipeline I built was the input that made all of those features coherent. That integration — designing a component with its downstream consumers in mind — is what made the architecture hold together.
