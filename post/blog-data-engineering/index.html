Data Engineering on AWS — Pipelines, Orchestration & Scale

April 5, 2026

There's a version of data engineering that's mostly about moving files from one place to another. What I built was closer to an operating system for data — a set of interconnected services that could accept data from any source, process it through a full ML pipeline, and deliver a structured, queryable catalog, entirely without manual intervention, deployed across both a shared cloud environment and directly inside enterprise client AWS accounts.

Everything ran on AWS. Orchestration ran on a managed Airflow service. Compute for large-scale profiling ran on a managed distributed processing service. Model inference ran on a cloud ML platform. Storage and inter-service communication ran through object storage. Each piece had a specific reason for being chosen over alternatives.

Multi-Source Ingestion

The ingestion layer supported six different source types: tabular file uploads (CSV, Parquet, Excel), relational databases, cloud object storage, Azure SQL Server, a third-party accounting platform via OAuth2, and pre-signed URLs. Every source type produced the same normalized output before entering the pipeline downstream — so all processing logic was source-agnostic.

A few engineering details mattered here. Before creating any new data connection, the system checked whether an equivalent connection already existed (by comparing a hash of the credentials) to prevent duplicate onboarding. On refresh, it compared the current source contents against what had already been onboarded — adding new items, flagging removed ones, leaving existing datasets untouched. These two behaviors — deduplication and differential refresh — meant the system was safe to trigger repeatedly without accumulating garbage.

Big Data Profiling at Scale

For statistical profiling of enterprise datasets — which could reach hundreds of millions of rows — running processing on a single machine was not viable. I used a distributed computing approach on a managed cloud service to run profiling as an autoscaling job.

The profiling computed column-level completeness, cardinality, statistical summaries, and approximate quantile distributions using memory-efficient algorithms designed for large-scale data — estimating percentiles across very large datasets without loading them fully into memory. For clients with data governance requirements, the profiling output was written to cloud storage in a format directly queryable by other analytics services.

The pipeline waited for profiling completion through a sensor on the storage layer — polling until the output file appeared — rather than keeping a live connection to the compute job. This is the correct pattern for asynchronous cloud jobs: decouple the trigger from the result, and poll the durable artifact rather than the job status.

Orchestrating a Customer Entity Resolution Pipeline — Inside Client Infrastructure

One project took the data engineering work outside the platform entirely — into the private cloud environment of a large financial institution. Their requirement was strict: customer data could not leave their own environment. The pipeline ran in their infrastructure, not ours, and has been running in production in their environment for over three years.

The pipeline itself was a multi-stage Customer Entity Resolution system — resolving duplicate and fragmented customer records across systems into unified, canonical entities. My ownership was the orchestration layer: designing the DAGs that connected and sequenced every stage, scheduling daily delta runs, and building success and failure notification flows so the client's team always knew the pipeline's state without having to inspect it manually.

The triggering model was file-presence on object storage: each stage only started when the previous stage's output landed in the expected location. This made the pipeline naturally resumable — any stage could be rerun independently without restarting from the beginning, and no stage held open connections waiting for upstream work. For a pipeline running inside a client's environment where we had no direct debugging access, this design mattered: failure was isolated to a single stage, and the notification system surfaced exactly where it stopped.

Deploying orchestration into a client's environment rather than your own is operationally different in one critical way: you can't rely on being there to fix things. Every workflow had to be self-describing in its failures, every configuration had to be parameterized rather than hardcoded, and every notification had to give the client enough context to act on their own. The fact that this pipeline has run reliably for over three years without rebuilding is the proof that those design decisions were right.

OAuth2 Integration — A Production Auth Flow

Connecting to an external accounting platform via OAuth2 sounds straightforward until you actually implement it. The protocol has well-defined steps on paper. Production systems add layers of complexity that the spec doesn't cover: tokens expire, refresh tokens expire separately, users revoke access, environments get mixed up, and your auth flow needs to handle all of it gracefully without losing the client's data connection.

The access tokens were short-lived. Rather than waiting for a token to fail and then refreshing, the system proactively checked expiry on each connection use and refreshed the token if it was within a short window of expiring. This eliminated mid-request failures caused by tokens expiring between the check and the actual API call.

Refresh tokens have their own, longer expiry — but they do expire. The system tracked both timestamps and distinguished three distinct failure scenarios: an expired access token with a valid refresh token (automatically resolved), an access token where the refresh also fails (flagged for reauthorization with a specific error code), and a case where the refresh token itself has expired (requires the user to re-authorize from scratch). Each case produced a specific reason code rather than a generic auth failure — so the frontend could show the right message and prompt the right action.

The result was an auth integration that was silent when things were working and specific when they weren't — which is exactly what a production OAuth2 flow needs to be.

Why These Design Choices

The combination of event-driven orchestration, cloud-native compute, object storage as the universal inter-service layer, and robust multi-source ingestion wasn't assembled arbitrarily. Event-driven meant no polling and no wasted resources. Managed distributed compute meant autoscaling to client dataset size without provisioning. Object storage as the communication layer meant every intermediate artifact was durable, inspectable, and independently re-processable. Each design decision traded some complexity at setup time for reliability and operability at runtime. That trade-off is the right one for infrastructure running in environments you don't control.
